# Plan: Fix React Error #130 Console Errors

## Summary

**Error:** `Minified React error #130: Element type is invalid: expected a string (for built-in components) or a class/function but got: object`

**Pattern:** Two React Error #130 exceptions occur synchronously during initial app mount, caught by `ErrorBoundary` and logged as `[ErrorBoundary] Caught error: Object`. One stack shows Suspense at root boundary (4-item args array), the other shows a deeper nested Suspense (1-item args array), confirming **two separate invalid elements** thrown at mount time.

**Trigger:** Happens after `[Auth] State change event: INITIAL_SESSION`, when React tries to render the initial route (Landing Page on `/`).

---

## Root Cause (Three Contributing Factors)

### 1. `transformApp` Can Produce Invalid `ComponentApp` Objects from Stale Cached Data

**File:** `src/hooks/useApps.ts` lines 99-105

```ts
if (!forceRefresh) {
  const cachedApps = getCachedApps();
  if (cachedApps) {
    console.log("[useApps] Using cached data (skipping validation due to RLS issues)");
    setApps(cachedApps);
    setLoading(false);
    return;  // ← returns BEFORE supabase validation
  }
}
```

When Supabase RLS blocks access:
1. `useApps` falls back to stale cache without running `transformApp` at all
2. The data is old and may contain apps missing URLs, malformed iconNames, or fields that no longer match the current `ComponentApp` interface
3. No per-field validation occurs — the cached data is trusted as-is

Additionally, on fresh fetch, `transformApp(dbApp)` can produce objects with bad fields like:
- `iconName: undefined` when `dbApp.slug` is `undefined` (unknown data source)
- `url: /app/undefined` when `getCentralizedAppUrl(undefined)` is called
- `image: "https://images.unsplash.com/..."` (string, fine) vs icon elements stored as `React.ReactNode` vs icon name misuse

### 2. `appsData.ts` Stores `React.createElement()` Results as Static Data

**File:** `src/data/appsData.ts` (6466 lines, 116+ `React.createElement()` calls)

```ts
const rawAppsData: App[] = [
  {
    icon: React.createElement(Search),  // React node stored as data
    ...
  },
  // 460 more entries...
];
```

`DashboardToolsSection.tsx` and `DashboardPersonalizerSection.tsx` render these React elements directly using `{app.icon}` (which calls `React.isValidElement` + `React.cloneElement`). This works when the element is a proper React object, but:
- Bundle serialization can sometimes flatten or corrupt element references
- During SSR hydration mismatches or in concurrent rendering, the element's internal `$$typeof` can become stale
- When the same data is used across module boundaries (imported in dashboard but defined in data file), the element identity is preserved but its rendering contract may have shifted

### 3. `featuresData.ts` Same Pattern (280 `React.createElement`)

**File:** `src/data/featuresData.ts` (771 lines, `icon: React.createElement(...)` throughout)

Features components render `{feature.icon}` directly. Same corruption risk as `appsData.ts`.

### Error Flow

```
App mounts
  → Auth INITIAL_SESSION fires
  → Schedule root route render (Landing Page)
    → Suspense:AppGallerySection (first lazy section after SparkleBackground)
      → PublicAppGallery / AppGallerySection renders
        → useApps() returns apps
          → apps.map((app) => <SomeComponent key={app.id} iconNode={app.iconName invalid_or_undefined} />)
            → LazyIcon / icon rendering encounters invalid element/string → React #130
```

## Fix Plan

### Fix 1: Make `transformApp` Resilient to Null/Missing/Undefined Database Data

**File:** `src/hooks/useApps.ts`

- Add an `isValidAppData` guard function that validates all required fields of a `DatabaseApp` before transformation
- Wrap `transformApp(dbApp)` in a try/catch within the `.map()` call — if any individual record fails, log it and skip it instead of crashing
- Skip invalid apps rather than crashing the entire list
- Add computed `iconName` validation — ensure it's a non-empty string before passing to `LazyIcon`

```
Before: const transformedApps = data.map(transformApp);
After:  const transformedApps = data
          .filter(isValidAppData)
          .map(dbApp => safeTransform(dbApp))
          .filter(app => app != null);
```

### Fix 2: Replace `React.createElement()` Static Icons with Icon Name Strings

**Files:** `src/data/appsData.ts`, `src/data/featuresData.ts`

Replace all 396 `React.createElement(SomeIcon)` calls with simple string identifiers that `LazyIcon` can resolve at render time:

```ts
// Before (BROKEN as static data):
icon: React.createElement(Video)

// After (SAFE as data):
iconName: "video"
iconVariant: "filled"   // optional variant field
```

- `appsData.ts`: Replace 116 `React.createElement(...)` icon definitions with `iconName: "xxx"` strings
- `featuresData.ts`: Replace 280 `React.createElement(...)` icon definitions with `iconName: "xxx"` strings
- Update type definitions to include `iconName` field (additive — `icon` field at `React.ReactNode` type becomes deprecated/unused in favor of `iconName`)
- Consumers that do `{app.icon}` must change to `<LazyIcon name={app.iconName} />` — actually, since `appsData.ts` icons are currently used in:
  - **NOT** by `useApps`/Supabase apps (which use `iconName` string already) — those are fine
  - **ONLY** by dashboard components that import from `appsData.ts` — those need to switch from `{app.icon}` to `<LazyIcon name={app.iconName} />`

- Update `DashboardToolsSection.tsx` to render icons via `LazyIcon` instead of direct element injection
- Update `DashboardPersonalizerSection.tsx` to render icons via `LazyIcon`
- Update `AppDetailModal.tsx` to render icons via `LazyIcon` or `React.isValidElement(app.icon)` guard remains for backward compat

### Fix 3: Defensive Icon Rendering in Remaining Components

Components currently checking `React.isValidElement(app.icon)` and using `React.cloneElement` should be updated or remain as defensive fallback — this is safe already but should add a default fallback icon:

```tsx
// Current (safe but verbose):
{React.isValidElement(app.icon)
  ? React.cloneElement(app.icon as React.ReactElement, { className: "..." })
  : <Sparkles className="..." />

// After (simplified with LazyIcon):
<LazyIcon name={app.iconName || "sparkles"} className="..." />
```

### Fix 4: Add Data Integrity Validation to `useApps` Cache Read

**File:** `src/hooks/useApps.ts`

Add cache versioning and schema validation to prevent stale cache data from causing errors:

```ts
interface CacheData {
  data: ComponentApp[];
  timestamp: number;
  version: string;  // "1" = current schema
}

// Reject cache if version is missing or stale
const CACHE_VERSION = "1";
```
