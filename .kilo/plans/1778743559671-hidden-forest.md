# Admin User Access Management System - Enhancement Plan

## Status: PLANNING

## Goal
Enhance the existing admin system to allow super admins to easily grant/revoke user access to all apps and bundles with simple switches and bulk operations. The system should be intuitive, connected to Supabase, and support future user growth.

## Current State Analysis

### Existing Infrastructure
- **Admin Dashboard** (`src/pages/AdminDashboard.tsx`): Tabbed interface with multiple management sections
- **AdminUsersManagement** (`src/components/admin/AdminUsersManagement.tsx`): Manages users with:
  - User list with active/inactive toggles
  - Individual user "Manage Apps" modal with per-app toggles
  - Select All / Deselect All buttons
  - Feature access management
  - Bulk user creation via CSV
- **Backend Functions** (`supabase/functions/`):
  - `admin-users/index.ts`: Handles user CRUD, app access GET/POST/DELETE
  - `admin-apps/index.ts`: Manages apps
  - Full support for bulk operations already in place
- **Database**:
  - `user_roles`: Stores user roles (super_admin, admin, user)
  - `user_app_access`: Tracks which apps each user can access
  - `apps`: Contains ~102 apps across 12 categories
- **Bundle System**: Defined in `src/data/bundleData.ts` with 12 category bundles (10-16 apps each)

### What's Already Working
1. User creation, deletion, activation toggles
2. Per-user app access management with modal
3. Bulk operations support in backend (app_slugs array)
4. Apps grouped by category in modal
5. Optimistic UI updates with error recovery

### What's Missing (User Pain Points)
1. No quick "Grant All Apps" toggle per user (must manually toggle each app)
2. No bundle-based access control (grant all apps in a bundle with one click)
3. No visual grouping of apps by category on user cards
4. No bulk operations across multiple users
5. No quick search/filter for users
6. App management requires opening modal for each user

## Technical Design Decisions

### State Management
- Keep using React local state in AdminUsersManagement (fine-grained, no global needed)
- Add React Query or SWR if data fetching becomes complex (currently using fetch+cache)
- Consider context for shared state across admin tabs (optional)

### Performance
- Virtual scrolling for user list if > 1000 users
- Debounce search input (300ms)
- Batch API calls for bulk operations (100 users max per batch)
- Lazy load bundle data from static import (already tree-shakeable)

### RBAC (Role-Based Access Control)
- super_admin: full access to all features including user management
- admin: can manage users but cannot assign super_admin role
- UI should hide/show Grant All based on role (currently both admin and super_admin can access)
- Consider adding more granular permissions (future)

### Error Handling
- Keep existing error boundaries
- Add retry logic with exponential backoff (already in useUserAccess hook)
- Show clear error messages with recovery actions
- Transactional integrity: bundle grants should be all-or-nothing

### UX/UI Consistency
- Use existing design system: Tailwind + Radix UI + Lucide icons
- Follow existing AdminUsersManagement patterns
- Use same toggle switch component (Radix Switch)
- Use same notification system (currently using success/error state banners)

## Database Schema Review

### Existing Tables (sufficient)
```sql
user_roles (user_id, role, tenant_id)
user_app_access (user_id, app_slug, access_type, is_active, granted_at, expires_at)
apps (id, slug, name, category, is_active)
```

### No Schema Changes Needed
- Bundle info is static (frontend bundleData.ts)
- user_app_access can store individual app grants
- Bundle grants = multiple rows in user_app_access

### Potential Future Additions
- `user_bundle_grants` table to track bundle-level grants (instead of expanding to individual apps)
- `user_note` table for admin annotations
- `user_import_batches` for tracking bulk imports

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| 102 API calls on "Grant All" overloads server | High | Use bulk upsert (already supported), batch in chunks of 50 |
| Race conditions with multiple admins | Medium | Optimistic locking, refresh on conflict |
| Partial failure (some apps grant, some fail) | Medium | Transaction-like behavior, rollback on error |
| Bundle definition changes break existing grants | Low | Bundle data is static; if apps added to bundle later, admin must regrant |
| UI clutter from too many toggles | Medium | Use collapsible sections, progressive disclosure |
| Privacy concerns: exposing all app slugs | Low | Already exposed in existing modal, RLS protects |
