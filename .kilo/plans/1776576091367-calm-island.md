# Complete Bulk User Import Plan

## Current State Analysis
- ✅ Database schema is ready with CSV import tables
- ✅ Product catalog has 26 active products with app access mappings  
- ✅ `all_users_for_import.csv` exists (transformed data from multiple CSVs)
- ⚠️ Previous import attempts failed due to product name matching issues
- ⚠️ 9,439 user records exist but all marked as "failed" status

## Import Process Steps

### Step 1: Verify Environment and Prerequisites
- Confirm Supabase local development is running
- Verify `.env` file has required credentials (`VITE_SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`)
- Check Node.js 18+ is available
- Ensure `bulk-create-auth-users.js` script exists in project root

### Step 2: Dry Run Import (--dry-run)
- Run `node bulk-create-auth-users.js --dry-run` 
- This will:
  - Load existing auth users and product catalog
  - Parse `all_users_for_import.csv` 
  - Show what would be created without making changes
  - Identify potential product matching issues
- Review dry run output for:
  - Number of users to create vs existing
  - Product matching success rate
  - Any errors or warnings

### Step 3: Execute Live Import
- If dry run looks good, run `node bulk-create-auth-users.js`
- This will:
  - Create Supabase Auth users (with random secure passwords)
  - Auto-confirm email addresses
  - Create/update profile records
  - Attempt to match products and grant app access
  - Track all operations in csv_imports and import_user_records tables

### Step 4: SQL Fallback for Access Grants
- If Step 3 fails to grant access due to platform constraints or missing products:
- Run `supabase db execute -f grant-bulk-product-access.sql`
- This SQL script uses fuzzy matching to:
  - Match CSV product names to products_catalog entries
  - Create missing purchases with valid platform ('stripe')
  - Grant app access via user_app_access table
  - Update import_user_records status to 'processed'

### Step 5: Verification and Cleanup
- Check import results in Supabase dashboard
- Verify user counts in auth.users
- Confirm app access distribution across apps
- Test login with sample users (passwords are not stored)
- Review any remaining failed records

## Risk Mitigation
- **Dry run first**: Always test with --dry-run before live import
- **Idempotent operations**: Scripts handle duplicate users gracefully
- **SQL fallback**: If JavaScript import fails, SQL can complete access grants
- **No destructive actions**: Import only adds data, doesn't delete existing records

## Expected Outcomes
- ~4,414 unique users (based on CSV data)
- Auth users created with secure random passwords
- Product-based app access granted according to catalog mappings
- Comprehensive import tracking and audit trail

## Contingency Plans
- If product matching fails extensively: Manually add missing products to catalog first
- If platform constraint errors: Use SQL fallback which uses 'stripe' platform
- If import partially succeeds: Re-run scripts (they handle duplicates safely)

## Success Criteria
- ✅ All users from CSV have auth accounts
- ✅ Profiles created for all users  
- ✅ Product purchases recorded
- ✅ App access granted based on products
- ✅ No failed import records remaining
- ✅ Users can log in to appropriate apps
